Vulnérabilités de Sécurité

8 issues
Critique
S1 — Mot de passe stocké en clair dans la classe Utilisateur
Source : Diagramme de classes → classe Utilisateur
›

La classe Utilisateur possède l'attribut String motDePasse. Pourtant l'architecture utilise Firebase Auth pour l'authentification. Cela signifie soit que les mots de passe sont dupliqués et stockés en base MongoDB, soit que le diagramme est incohérent. Dans le premier cas, c'est une faille critique : si MongoDB est compromis, tous les mots de passe sont exposés.

⚠️ ImpactViolation RGPD/CNDP directe. Exposition totale des credentials de tous les utilisateurs en cas de dump de base. Contredit entièrement l'utilisation de Firebase Auth.
✅ CorrectionSupprimer le champ motDePasse du modèle entièrement. Firebase Auth est le provider d'identité — le backend ne doit jamais manipuler les mots de passe. Seul le firebaseId (UID) doit relier les deux systèmes.
Critique
S2 — Élévation de privilèges via le champ role dans USERS
Source : DB Schema → table USERS · API Gateway
›

Le champ role: CANDIDATE | RECRUITER | ADMIN est stocké directement dans la collection USERS de MongoDB. Si le backend lit ce rôle depuis la DB pour autoriser les actions (notamment les routes /admin/*), un attaquant ayant accès à la DB (injection, SSRF, misconfiguration) peut modifier son rôle et devenir ADMIN.

De plus, l'API expose PUT /users/me — si ce endpoint permet de modifier le champ role, n'importe quel utilisateur peut s'auto-promouvoir.

⚠️ ImpactCompromission complète de la plateforme. Un candidat peut accéder au backoffice admin, valider des entreprises, bloquer des offres, voir toutes les données.
✅ CorrectionLes rôles ADMIN doivent être gérés via Firebase Custom Claims (immutables côté client). Le champ role en DB ne doit être qu'un cache en lecture seule. PUT /users/me doit explicitement blacklister les champs role, plan, isActive.
Critique
S3 — Absence de rate limiting sur les endpoints AI et Auth
Source : API Gateway → POST /ai/cv/analyze, POST /auth/register, POST /auth/login
›

L'API REST expose des endpoints coûteux sans limite de débit visible :

  • POST /ai/cv/analyze — appelle un LLM/ML, très coûteux
  • POST /ai/profile/optimize — idem
  • POST /auth/login — vecteur de brute-force
  • POST /applications — spam de candidatures
⚠️ ImpactDoS économique (facture IA explosive), brute-force des comptes, spam de candidatures fictives qui polluent les métriques des recruteurs. Un seul acteur malveillant peut vider le budget IA de la plateforme en minutes.
✅ CorrectionImplémenter un rate limiter à l'API Gateway (ex : Spring Boot + Bucket4j, ou au niveau Kubernetes Ingress). Pour les routes IA : max 10 analyses/heure/utilisateur pour Premium, 0 pour Free. Pour /auth/login : max 5 tentatives/15min par IP.
Critique
S4 — Données de paiement sensibles en texte libre dans PAYMENTS
Source : DB Schema → table PAYMENTS
›

La table PAYMENTS stocke : amount, currency, method (string libre), status, paidAt. Le champ method non structuré laisse entendre que des informations de paiement pourraient y être stockées. Aucune mention de tokenisation ou de prestataire PCI-DSS (Stripe, PayPal, CMI...).

⚠️ ImpactNon-conformité PCI-DSS. Si des numéros de carte ou coordonnées bancaires transitent par le backend et sont loggés, c'est une violation légale majeure, passible de sanctions.
✅ CorrectionUtiliser un prestataire de paiement certifié (Stripe, PayDunya, CMI pour le Maroc). Ne stocker que le paymentIntentId ou le token de transaction. Jamais de données brutes de carte en DB.
Critique
S5 — Accès non autorisé aux CV (cvUrl exposée sans contrôle)
Source : DB Schema → CANDIDATE_PROFILES.cvUrl · API POST /candidate-profiles/me/cv
›

Le champ cvUrl dans CANDIDATE_PROFILES pointe vers un fichier stocké (Firebase Storage ou autre). Si l'URL est un lien direct sans token signé et avec une durée de vie illimitée, n'importe qui ayant accès à la DB (ou interceptant une réponse API) peut télécharger le CV d'un candidat sans autorisation.

⚠️ ImpactViolation de la vie privée des candidats. Un recruteur non autorisé, ou pire un candidat, pourrait accéder aux CV de tous les autres utilisateurs.
✅ CorrectionUtiliser des Firebase Storage Signed URLs avec expiration courte (ex: 15 min), générées à la demande par le backend après vérification des droits. Ne jamais exposer l'URL permanente du fichier.
Élevé
S6 — Injection NoSQL possible via string[] (requiredSkills, skills)
Source : DB Schema → OFFERS.requiredSkills, CANDIDATE_PROFILES.skills
›

Les champs requiredSkills: string[] et skills: string[] sont des tableaux de chaînes libres en MongoDB. Si le backend construit des requêtes de matching en interpolant directement ces valeurs sans sanitisation, un attaquant peut injecter des opérateurs MongoDB ($where, $gt, etc.).

// Vulnérable
db.offers.find({ requiredSkills: { $in: userInput } })
// userInput = [{ "$gt": "" }] → retourne tout
✅ CorrectionValider et sanitiser toutes les entrées utilisateur. Utiliser un ODM (Mongoose) avec des schemas stricts. Désactiver les opérateurs MongoDB dans les inputs utilisateur. Utiliser des paramètres typés, pas de construction de requêtes par string interpolation.
Élevé
S7 — Absence de vérification email avant activation du compte
Source : Diagramme de séquence — Scénario 1 (Inscription)
›

Le flow d'inscription dans le diagramme de séquence va directement de "Créer compte Firebase" → "Token Firebase" → "Enregistrer utilisateur" → "Compte créé avec plan Free" sans aucune étape de vérification d'email. Un acteur malveillant peut créer des centaines de faux comptes avec des emails inexistants.

⚠️ ImpactSpam de candidatures fictives, pollution des statistiques, abus des fonctionnalités Free, contournement potentiel des rate limits basés sur le compte.
✅ CorrectionFirebase Auth supporte nativement la vérification d'email (sendEmailVerification()). Le backend doit vérifier firebaseUser.emailVerified = true avant d'activer le compte. Bloquer l'accès aux fonctionnalités clés jusqu'à vérification.
Moyen
S8 — Données sensibles des candidats dans AI_RESULTS sans chiffrement
Source : DB Schema → table AI_RESULTS
›

La table AI_RESULTS stocke extractedSkills: string[], recommendation: string, details: object. Ces données sont le résultat de l'analyse du CV — potentiellement des informations très personnelles (compétences, niveau d'études, expériences). Aucun mécanisme de chiffrement au repos n'est mentionné.

✅ CorrectionChiffrement au repos MongoDB (MongoDB Encrypted Storage Engine ou chiffrement applicatif avec AES-256 pour les champs details et recommendation). Définir une politique de rétention (ex: TTL de 90 jours sur AI_RESULTS).

Scalabilité & Performance

7 issues
Critique
P1 — Table ANALYTICS : single point of contention pour tous les compteurs
Source : DB Schema → table ANALYTICS
›

La table ANALYTICS contient un seul document avec des compteurs numériques : totalUsers, totalCandidates, totalRecruiters, totalOffers, totalApplications, totalPremiumUsers. Chaque inscription, candidature, publication d'offre ou abonnement Premium nécessite une mise à jour de ce document unique.

Sous charge (ex: 1000 inscriptions simultanées), MongoDB doit acquérir un verrou sur ce document pour chaque écriture → sérialisation complète des opérations → goulot d'étranglement catastrophique.

⚠️ ImpactÀ partir de quelques centaines d'utilisateurs concurrents, les temps de réponse de toute l'application augmentent exponentiellement. MongoDB peut saturer sur ce seul document.
✅ CorrectionUtiliser MongoDB $inc avec des compteurs shardés ou mieux : calculer les statistiques en temps réel avec des agrégations MongoDB, ou utiliser un pipeline de streaming (Kafka/Flink) pour accumuler les métriques. Redis Counters pour les compteurs temps-réel.
Élevé
P2 — Analyse IA synchrone bloquante dans le flow utilisateur
Source : Séquence Scénario 2 — "Candidat Premium utilise l'IA"
›

Le scénario 2 montre que lorsqu'un candidat demande des recommandations IA, le backend fait en synchrone : lire le CV, lire les offres, calculer le matching, enregistrer les résultats, puis retourner la réponse. Une analyse LLM peut prendre 5–30 secondes. La connexion HTTP reste ouverte pendant tout ce temps.

⚠️ ImpactAvec 100 utilisateurs Premium simultanés demandant une analyse, le thread pool Spring Boot sature. L'API devient inaccessible pour tous les autres utilisateurs (CANDIDATS FREE, recruteurs).
✅ CorrectionRendre l'analyse IA asynchrone via une queue (RabbitMQ, Kafka, ou Firebase Cloud Tasks). Le client reçoit un jobId immédiatement (202 Accepted), puis poll ou reçoit une notification push quand le résultat est prêt. Pattern Job Queue.
Élevé
P3 — Absence de cache pour les listings d'offres (GET /offers)
Source : API Gateway → GET /offers
›

Le endpoint GET /offers (listing public des offres) n'a aucune couche de cache visible dans l'architecture. C'est typiquement l'endpoint le plus consulté d'une plateforme (rafraîchi à chaque visite, même pour les non-connectés). Chaque requête fait un full-scan MongoDB.

⚠️ ImpactAvec 10 000 utilisateurs actifs, la collection OFFERS est interrogée des dizaines de milliers de fois par heure. Sans index ni cache, MongoDB s'effondre.
✅ Correction1) Redis cache avec TTL de 60 secondes pour GET /offers (invalidé lors de chaque POST/PUT/DELETE offer). 2) Index MongoDB sur status, type, location, publishedAt. 3) Pagination obligatoire (limit/offset ou cursor-based) — jamais de retour de toute la collection.
Élevé
P4 — Croissance non bornée de AI_RESULTS sans politique de rétention
Source : DB Schema → table AI_RESULTS
›

Chaque appel IA (analyse CV, recommandation d'offre, optimisation profil) crée un nouveau document dans AI_RESULTS. Un utilisateur Premium actif peut générer 5–10 analyses par jour. Avec 1000 Premium : ~1M documents/mois. Le champ details: object peut contenir des données volumineuses.

✅ CorrectionAjouter un TTL index MongoDB sur createdAt (ex: 90 jours). Compresser le champ details ou le stocker séparément (GridFS ou Firebase Storage). Ne garder que le dernier résultat par (userId, type).
Élevé
P5 — Architecture monolithique malgré la mention de microservices
Source : Structure du projet → "Backend Spring Boot REST API"
›

Le diagramme "Structure du projet" montre un seul bloc "Backend Spring Boot REST API" qui gère Auth, Users, Profiles, Companies, Offers, Applications, Subscriptions, AI, Notifications, et Admin. Pourtant le projet mentionne Kubernetes et Docker Compose. Déployer un monolithe sur Kubernetes ne donne pas les bénéfices des microservices.

⚠️ ImpactUne montée en charge du service IA oblige à scaler toute l'application. Un bug dans le module Admin peut crasher le module candidatures. Impossible de mettre à jour les modules indépendamment.
✅ CorrectionSi microservices souhaités, séparer au minimum : auth-service, offer-service, application-service, ai-service, notification-service. Sinon, rester sur un monolithe honnête mais bien structuré en modules Spring.
Moyen
P6 — Matching score recalculé à chaque candidature sans invalidation
Source : DB Schema → APPLICATIONS.matchingScore + AI_RESULTS
›

Le matchingScore dans APPLICATIONS est calculé par le service IA. Mais si un candidat met à jour son profil après avoir postulé, le score devient périmé. Aucun mécanisme d'invalidation ou de recalcul n'est visible dans l'architecture.

✅ CorrectionStocker la date de calcul du score (scoreComputedAt). Lors de chaque mise à jour du profil candidat, invalider ou re-queue les scores de ses candidatures actives. Afficher côté UI "Score basé sur le profil du [date]".
Moyen
P7 — Absence de pagination définie dans l'API
Source : API Gateway — tous les endpoints GET collection
›

Les endpoints GET /offers, GET /applications, GET /notifications, GET /admin/users ne montrent aucun paramètre de pagination (page, limit, cursor). Sans pagination, une réponse peut contenir des milliers d'objets JSON, saturant la mémoire du client et du serveur.

✅ CorrectionImposer une pagination obligatoire avec une taille max par défaut (ex: 20 items) et un maximum absolu (100). Utiliser le pattern cursor-based pour les collections qui changent souvent (notifications, candidatures).

Incohérences de Logique Métier

7 issues
Élevé
L1 — Double source de vérité pour le plan (USERS.plan vs SUBSCRIPTIONS)
Source : DB Schema → USERS.plan + table SUBSCRIPTIONS
›

Le champ plan: FREE | PREMIUM existe dans deux endroits : la table USERS ET la table SUBSCRIPTIONS. Le séquence diagramme montre "Lire plan utilisateur" → "Plan Premium" depuis la DB. Si les deux valeurs divergent (ex : USERS.plan=PREMIUM mais SUBSCRIPTIONS.status=EXPIRED), laquelle fait foi ?

Scénario concret : un utilisateur Premium annule son abonnement. SUBSCRIPTIONS.status passe à CANCELLED. Mais si USERS.plan n'est pas mis à jour en même temps (race condition, erreur réseau), l'utilisateur accède aux fonctionnalités Premium indéfiniment.

⚠️ ImpactFuite de revenus (accès Premium gratuit), ou au contraire des utilisateurs payants bloqués (si la désynchronisation va dans l'autre sens). Impossible à déboguer sans tracer les deux tables.
✅ CorrectionSupprimer plan de USERS. La source de vérité unique est SUBSCRIPTIONS (jointure nécessaire mais cohérence garantie). Créer une vue/helper isUserPremium(userId) qui vérifie SUBSCRIPTIONS WHERE userId=? AND status=ACTIVE AND expirationDate > NOW().
Élevé
L2 — Offre publiable par une entreprise non validée
Source : DB Schema → COMPANIES.validationStatus + OFFERS.status · Use Case
›

La table COMPANIES a un champ validationStatus: PENDING | APPROVED | REJECTED. La table OFFERS a un champ status: DRAFT | PUBLISHED | ARCHIVED | BLOCKED. Il n'y a aucune contrainte visible dans le schéma DB ou la logique métier qui empêche un recruteur de publier une offre (POST /offers) pour une entreprise en statut PENDING ou REJECTED.

⚠️ ImpactDes entreprises frauduleuses peuvent publier des offres avant validation admin, exposant les candidats à des arnaques.
✅ CorrectionDans le service Offer, avant tout PUBLISH, vérifier que company.validationStatus == APPROVED. Retourner HTTP 403 avec message explicite sinon. Ajouter cette règle dans le diagramme de classes comme précondition de Offre.publier().
Élevé
L3 — Champs redondants incohérents dans SUBSCRIPTIONS (active vs status)
Source : DB Schema → table SUBSCRIPTIONS
›

La table SUBSCRIPTIONS possède à la fois :
— boolean active
— status: ACTIVE | EXPIRED | CANCELLED
Ces deux champs peuvent se contredire : active = true avec status = EXPIRED est un état incohérent mais parfaitement possible si la mise à jour est partielle.

✅ CorrectionSupprimer le champ boolean active — il est redondant. Le statut réel s'infère de status == ACTIVE AND expirationDate > NOW(). Ajouter un index sur expirationDate et un job cron qui passe les abonnements expirés à EXPIRED automatiquement.
Moyen
L4 — recruiterId redondant dans APPLICATIONS (dénormalisation risquée)
Source : DB Schema → table APPLICATIONS
›

La table APPLICATIONS stocke offerId, candidateId ET recruiterId. Or, le recruteur est déjà identifiable via : APPLICATIONS.offerId → OFFERS.companyId → COMPANIES.recruiterId. Stocker recruiterId directement crée une dénormalisation : si le recruteur associé à une entreprise change, les anciennes candidatures pointent vers un recruteur obsolète.

✅ CorrectionSupprimer recruiterId de APPLICATIONS. Le résoudre à la query via jointure. Si la performance est un souci, le mettre dans un champ calculé non persisté ou dans un index.
Moyen
L5 — Aucune date d'expiration sur les offres INTERNSHIP
Source : DB Schema → table OFFERS
›

La table OFFERS a les champs duration et publishedAt mais aucun expiresAt ou applicationDeadline. Une offre de stage publiée en janvier reste techniquement PUBLISHED en décembre. Les candidats peuvent postuler à des offres pour des stages déjà pourvus.

✅ CorrectionAjouter applicationDeadline: date (optionnel mais affiché) et autoArchiveAfter: number (jours). Mettre en place un job cron quotidien qui passe les offres dont applicationDeadline < TODAY en statut ARCHIVED.
Moyen
L6 — Favoris non invalidés si l'offre est archivée ou bloquée
Source : DB Schema → table FAVORITES
›

La table FAVORITES stocke userId + offerId sans vérification de l'état de l'offre. Si une offre passe à ARCHIVED ou BLOCKED, les entrées FAVORITES correspondantes restent valides. Côté UI, le candidat verra toujours l'offre dans ses favoris, mais ne pourra plus y postuler — sans explication claire.

✅ CorrectionLors du fetch des favoris, faire une jointure sur OFFERS.status et filtrer les offres non-PUBLISHED. Ou, lors du passage d'une offre à ARCHIVED/BLOCKED, supprimer les FAVORITES correspondants (trigger ou event). Afficher côté UI une bannière "Cette offre n'est plus disponible".
Info
L7 — Flux de paiement Premium non sécurisé lors de l'inscription
Source : Wireframe screen 02 + Séquence Scénario 1
›

Le wireframe d'inscription (écran 02) présente le choix du plan (Free / Premium IA) avant la création du compte. Pourtant, le diagramme de séquence montre le Premium comme un bloc opt (optionnel) après création du compte. Cette incohérence entre UI et logique peut mener à des cas où un utilisateur choisit Premium, crée son compte, mais le paiement échoue — laissant le compte en état Free sans message d'erreur clair.

✅ CorrectionAligner le wireframe et la séquence. Recommander : créer le compte Free d'abord, puis rediriger vers un flow de paiement dédié (page Upgrade). Ne jamais activer le plan Premium avant confirmation de paiement réussie (webhook du prestataire).
